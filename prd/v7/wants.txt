# cyolo PRD

## 1. 개요

`cyolo`는 Claude Code의 멀티 계정 관리와 설정 파일 유지보수를 위한 단일 바이너리 CLI 유틸리티다. 기존 `cyolo` zsh 함수를 대체하면서 두 가지 새 기능을 추가한다:

1. **프로필 기반 계정 전환** — 폴더별로 서로 다른 Claude Code 계정을 자동 사용
2. **설정 파일 정리(diet)** — `~/.claude.json`과 세션 히스토리의 누적된 데이터 정리

### 1.1 배경

Claude Code는 `CLAUDE_CONFIG_DIR` 환경변수로 자격 증명 저장 위치를 지정할 수 있다. 이를 계정별로 분리하면 로그아웃 없이 여러 계정을 병행 사용할 수 있지만:

- 매번 환경변수를 수동 설정하거나 alias를 외워야 한다
- 새 프로필을 만들 때마다 npx로 설치한 skills, MCP 설정, CLAUDE.md 등을 다시 설정해야 한다
- `~/.claude.json`은 시간이 지나며 삭제된 프로젝트 기록까지 계속 쌓인다

`cyolo`는 이 세 가지 문제를 해결한다.

### 1.2 대상 사용자

개인 용도. 작성자 본인이 회사/개인 등 여러 Claude Code 계정을 사용하고, 설정(skills, commands, MCP 등)은 계정 간 공유하고 싶은 경우.

### 1.3 핵심 해결 과제

- **로그인/로그아웃 반복 제거**
- **설정 공유** — 한 번 설치한 skill, 한 번 작성한 CLAUDE.md를 모든 계정에서 사용
- **설정 파일 비대화 방지** — 삭제된 프로젝트의 잔여 데이터 정리

## 2. 기존 `cyolo` 함수 대체

현재 사용 중인 zsh 함수:

```bash
cyolo() {
  if [[ "$1" == "update" ]]; then
    claude update
    return
  fi
  claude --dangerously-skip-permissions "$@"
}
```

`cyolo` 바이너리는 이 동작을 **100% 보존**하면서 프로필/diet 기능을 추가한다. 프로필 기능을 쓰지 않아도 기존 사용 경험과 동일하게 동작한다.

## 3. 기본 실행 동작

```
cyolo                      → 프로필 감지 + claude --dangerously-skip-permissions 실행
cyolo <args...>            → 프로필 감지 + claude --dangerously-skip-permissions <args...>
cyolo update               → claude update 실행 (프로필 감지 없음)
cyolo profile <...>        → 프로필 관리 서브커맨드
cyolo diet <...>           → 설정 파일 정리 서브커맨드
```

- `update`, `profile`, `diet` 외의 모든 인자는 claude에 투명하게 전달
- 기본 설치 상태(프로필 등록 없음)에서는 기존 함수와 완전히 동일하게 동작

### 3.1 프로필 결정 우선순위

실행 시점에 다음 순서로 프로필 결정:

1. **`.claude-profile.json`** — 현재 폴더에서 상위로 walk-up하며 탐색, 발견 시 해당 프로필 사용
2. **default 프로필** — `cyolo profile default`로 등록된 것
3. **없음** — `CLAUDE_CONFIG_DIR`을 unset한 채 claude 실행 (`~/.claude` 사용, claude 기본 동작)

3번에서 경고/에러 없이 조용히 기본 동작으로 넘어간다. 기존 `cyolo` 함수와 동일한 경험.

## 4. 프로필 관리

### 4.1 `.claude-profile.json` 스키마

두 가지 방식 모두 지원, `name`이 우선.

**이름 참조 (권장):**
```json
{ "name": "work" }
```

**인라인:**
```json
{ "config_dir": "~/.claude-custom" }
```

`name`이 지정되었는데 전역 config에 등록되지 않은 이름이면 에러 출력 후 종료.

### 4.2 Walk-up 탐색

Git이 `.git`을 찾는 방식처럼 cwd부터 루트까지 `.claude-profile.json`을 탐색. 프로젝트 루트에 하나 두면 모든 서브디렉토리에서 작동.

### 4.3 프로필 디렉토리 구조 (symlink 기반 공유)

**핵심 원칙**: `~/.claude/`를 "원본"으로 두고, 다른 프로필은 설정 파일을 symlink로 참조한다. npx로 설치하는 skills가 `~/.claude/skills/`에 저장되기 때문에, symlink 방식이어야 한 번 설치한 skill을 모든 프로필에서 쓸 수 있다.

```
~/.claude/                          ← 원본 (personal 계정으로도 사용 가능)
├── .credentials.json               (personal 자격 증명, 고유)
├── projects/                       (personal 세션 히스토리, 고유)
├── statsig/                        (고유)
├── todos/                          (고유)
├── shell-snapshots/                (고유)
├── CLAUDE.md                       ← 원본
├── settings.json                   ← 원본
├── settings.local.json             ← 원본
├── commands/                       ← 원본
├── skills/                         ← 원본 (npx 설치 타겟)
└── agents/                         ← 원본

~/.claude-work/                     ← cyolo가 생성
├── .credentials.json               (work 자격 증명, 고유)
├── projects/                       (work 세션 히스토리, 고유)
├── statsig/                        (고유)
├── todos/                          (고유)
├── shell-snapshots/                (고유)
├── CLAUDE.md            → ~/.claude/CLAUDE.md
├── settings.json        → ~/.claude/settings.json
├── settings.local.json  → ~/.claude/settings.local.json
├── commands/            → ~/.claude/commands
├── skills/              → ~/.claude/skills
└── agents/              → ~/.claude/agents
```

### 4.4 기본 공유 대상 (6개)

- `CLAUDE.md`
- `settings.json`
- `settings.local.json`
- `commands/`
- `skills/`
- `agents/`

### 4.5 고유 대상 (공유 안 함)

- `.credentials.json` (Linux/Windows, macOS는 Keychain)
- `projects/` — 세션 히스토리
- `statsig/` — 분석 캐시
- `todos/`, `shell-snapshots/` — 런타임 상태

### 4.6 symlink 생성 규칙

`cyolo profile add` 시:

- `~/.claude/`에 **폴더**(`commands/`, `skills/`, `agents/`)가 없으면 → 빈 폴더로 먼저 생성한 뒤 symlink
- `~/.claude/`에 **파일**(`CLAUDE.md`, `settings.json`, `settings.local.json`)이 없으면 → symlink 생성 skip (빈 파일을 만들면 Claude Code가 오작동할 수 있음)
- 새 프로필 디렉토리에 이미 같은 이름의 파일/폴더가 있으면 → symlink 생성하지 않음 (덮어쓰지 않음, 경고 출력)

## 5. 프로필 관리 명령어

### 5.1 프로필 등록

```
cyolo profile add <name> [config-dir] [--default] [--no-share]
```

- `config-dir` 생략 시 `~/.claude-<name>` 자동 사용
- 디렉토리가 없으면 권한 `0700`으로 생성
- 기본 동작: `~/.claude/`의 6개 대상을 symlink로 연결
- `--default`: 이 프로필을 default로 지정
- `--no-share`: symlink 없이 완전 독립 프로필 (희귀 케이스)
- **예외**: `~/.claude`를 등록할 때는 symlink 생성하지 않음 (원본이므로)

예시:
```bash
cyolo profile add personal ~/.claude --default     # 원본을 personal로 등록
cyolo profile add work                              # ~/.claude-work 생성 + symlink
cyolo profile add client ~/.claude-client-a        # 경로 직접 지정
```

### 5.2 프로필 삭제

```
cyolo profile rm <name>
```

- config에서 제거, 실제 디렉토리는 **삭제하지 않음**
- 디렉토리 삭제는 사용자가 직접 `rm -rf ~/.claude-<name>` 실행

### 5.3 프로필 목록

```
cyolo profile list
```

출력 예시:
```
* personal → /Users/codingmax/.claude
  work     → /Users/codingmax/.claude-work
  client   → /Users/codingmax/.claude-client-a
```

### 5.4 default 프로필 설정

```
cyolo profile default <name>      → default 지정
cyolo profile default                 → 현재 default 출력
cyolo profile default --unset         → default 해제
```

### 5.5 프로필 파일 생성

```
cyolo profile init [name]
```

현재 폴더에 `.claude-profile.json` 생성. `name` 생략 시 default 프로필 이름 사용.

### 5.6 현재 상태 확인

```
cyolo profile current
```

출력 예시:
```
profile: work
config_dir: /Users/codingmax/.claude-work
source: /Users/codingmax/work/client-a/.claude-profile.json
```

### 5.7 symlink 재생성

```
cyolo profile link <name>
```

이미 등록된 프로필에 대해 symlink만 다시 생성. 공유 대상을 나중에 추가하거나 symlink가 깨진 경우 사용.

## 6. diet 명령어 (설정 파일 정리)

`~/.claude.json`과 `~/.claude/projects/`에 쌓인 불필요한 데이터를 정리한다.

### 6.1 정리 대상

#### (A) Orphaned 프로젝트 — 기본 동작

`~/.claude.json`의 `projects` 객체에서 **파일시스템에 더 이상 존재하지 않는** 경로의 항목 제거. 해당 경로의 `~/.claude/projects/<encoded-path>/` 세션 폴더도 함께 제거.

예: `/Users/codingmax/Private/labs/test-bot` 폴더가 삭제되었다면 해당 키 삭제.

#### (B) Stale 프로젝트 — 옵션

존재하지만 N일 이상 활동이 없는 프로젝트의 히스토리 제거 (프로젝트 항목 자체는 보존, `history` 배열만 비움).

#### (C) 캐시 디렉토리 — 옵션

```
~/.claude/statsig/          # 분석 캐시
~/.claude/shell-snapshots/  # 쉘 스냅샷
~/.claude/file-history/     # 파일 편집 히스토리
```

### 6.2 명령어

```
cyolo diet                              # dry-run, orphaned 리포트만
cyolo diet --apply                      # orphaned 실제 제거
cyolo diet --stale-days 90              # 90일 이상 비활성 포함 (dry-run)
cyolo diet --stale-days 90 --apply      # 실제 제거
cyolo diet --cache                      # 캐시 디렉토리 포함
cyolo diet --all                        # 모든 등록 프로필 대상
cyolo diet --profile work               # 특정 프로필만
```

### 6.3 dry-run 출력 형식

```
$ cyolo diet
cyolo diet — analyzing /Users/codingmax/.claude

~/.claude.json:                          1.2 MB  (6840 lines)
  ├─ orphaned projects (5):              980 KB  (removable)
  │   ├─ /Users/codingmax/Private/labs/test-bot      320 KB
  │   ├─ /Users/codingmax/tmp/experiment             210 KB
  │   └─ ... 3 more
  └─ active configuration:               220 KB  (keep)

~/.claude/projects/:                      847 MB
  └─ orphaned session folders (5):       623 MB  (removable)

Total reclaimable: 624 MB

Run with --apply to proceed.
```

### 6.4 안전장치

- **기본 dry-run**: `--apply` 없으면 아무것도 지우지 않음
- **자동 백업**: `--apply` 시 `~/.claude.json`을 `~/.claude.json.backup-<YYYYMMDDHHMMSS>`로 복사
- **실행 중 claude 감지**: `pgrep -f "claude"`로 프로세스 확인, 실행 중이면 경고 출력 후 중단 (`--force`로 우회 가능)
- **Atomic write**: 임시 파일에 쓴 뒤 `fsync` + `rename`으로 교체 (중간에 죽어도 파일 손상 방지)
- **백업 보존**: 최근 5개의 백업만 유지, 오래된 건 자동 삭제

### 6.5 프로필 고려

`--profile <name>` 또는 `--all` 없이 실행하면 **현재 활성 프로필만** 대상. 프로필 감지는 기본 실행 로직과 동일 (`.claude-profile.json` → default → `~/.claude`).

**주의**: symlink로 공유되는 구조에서는 `~/.claude.json`이 프로필별로 따로 존재하는지 구현 시 확인 필요 (섹션 10 참조).

## 7. 비기능 요구사항

### 7.1 구현

- **Rust**로 구현, 단일 바이너리
- 최소 의존성 (serde, serde_json, dirs 정도)
- 크로스 플랫폼 지원은 macOS/Linux만 (MVP)

### 7.2 전역 설정 파일

- 위치: `~/.cyolo/config.json`
- 권한: 파일 `0600`, 디렉토리 `0700`

형식:
```json
{
  "default": "personal",
  "profiles": {
    "personal": { "name": "personal", "config_dir": "~/.claude" },
    "work":     { "name": "work",     "config_dir": "~/.claude-work" }
  }
}
```

### 7.3 에러 처리

- `.claude-profile.json` JSON 파싱 실패 → 에러 출력 후 종료
- 등록되지 않은 프로필 이름 참조 → 친절한 안내
- symlink 생성 실패 (기존 파일 존재 등) → 경고 출력, 작업 계속
- diet 시 파일 잠금/권한 오류 → 에러 출력 후 중단

에러 메시지 예시:
```
cyolo: profile "work" not registered.
  Run: cyolo profile add work
```

### 7.4 설치

- `cargo install --path .`
- 또는 빌드된 바이너리를 `$PATH` 내 위치에 복사
- 기존 zsh 함수 제거 후 사용

## 8. 사용 시나리오

### 8.1 최초 셋업

```bash
# 기존 ~/.claude를 personal 프로필로 등록 (default 지정)
cyolo profile add personal ~/.claude --default

# 회사용 프로필 추가 (설정 자동 공유)
cyolo profile add work
# → ~/.claude-work 디렉토리 생성
# → ~/.claude/의 CLAUDE.md, settings, commands, skills, agents를 symlink

# 각 프로필로 최초 로그인
cyolo                          # personal 로그인 (이미 로그인되어 있음)
CLAUDE_CONFIG_DIR=~/.claude-work cyolo /login   # work 로그인
```

### 8.2 일상 사용

```bash
# 개인 프로젝트 — default(personal) 자동 적용
cd ~/hobby/my-app && cyolo

# 회사 프로젝트 — 폴더에 프로필 파일만 두면 끝
cd ~/work/client-a
cyolo profile init work
cyolo   # work 프로필로 실행

# 서브디렉토리에서도 walk-up으로 감지
cd ~/work/client-a/backend/services && cyolo
```

### 8.3 설정 변경

```bash
# npx로 skill 설치 → 모든 프로필에서 사용 가능 (symlink)
cd ~/.claude/skills && npx install-some-skill

# CLAUDE.md 편집 → 모든 프로필에 반영
$EDITOR ~/.claude/CLAUDE.md
```

### 8.4 정기 정리

```bash
# 월 1회 정도
cyolo diet                       # 먼저 dry-run으로 확인
cyolo diet --apply               # orphaned 제거

# 공간이 많이 필요하면
cyolo diet --stale-days 60 --cache --apply
```

## 9. Non-goals (MVP에서 제외)

- **쉘 hook 기반 자동 전환** — `cd` 시마다 자동 변경은 안 함, 명시적 `cyolo` 실행이 트리거
- **사용량 모니터링** — 5시간/7일 한도 추적 제외
- **실수 방지 경고** — 잘못된 계정으로 실행 중이라는 경고 제외
- **Git remote URL 기반 자동 매핑** — 제외
- **Windows 지원** — macOS/Linux만
- **symlink 진단 (`doctor`)** — 깨진 symlink 감지/복구 별도 명령어 제외 (`profile link`로 재생성)
- **공유 대상 커스터마이징** — 6개 공유 대상은 MVP에서 고정
- **macOS Keychain 조작** — 새 프로필에서 `/login` 다시 필요

## 10. 구현 시 검증 필요

### 10.1 `~/.claude.json`의 위치

`~/.claude.json` (사용자 레벨 MCP 설정)은 `~/.claude/` 디렉토리 바깥이다. `CLAUDE_CONFIG_DIR`을 바꾸면:

- **가설 A**: `~/.claude.json`은 변하지 않음 → 모든 프로필이 자동으로 공유, cyolo가 할 일 없음
- **가설 B**: `$CLAUDE_CONFIG_DIR/../.claude.json` 같이 따라감 → symlink 또는 별도 처리 필요

MVP 구현 시 실험으로 확인하고 대응. 가설 B인 경우 공유 대상에 `.claude.json` 추가.

### 10.2 diet의 대상 범위

`--profile` 옵션 사용 시 `~/.claude.json`을 프로필별로 따로 봐야 하는지, 공통으로 봐야 하는지 10.1 결과에 따라 결정.

### 10.3 symlink 이식성

symlink 대상이 절대 경로일 때와 상대 경로일 때의 동작 차이 확인. 프로필 디렉토리를 사용자가 나중에 이동할 가능성을 고려하면 상대 경로가 나을 수 있음.

## 11. 성공 기준

- 설치 후 계정 전환 때문에 `/login`을 치지 않는다
- 기존 `cyolo` 사용 습관이 그대로 유지된다
- 새 계정 연결이 `cyolo profile add <name>` + `/login` 한 번으로 끝난다
- 한 번 설치한 skill이 모든 프로필에서 자동으로 사용 가능하다
- 월 1회 `cyolo diet`로 설정 파일 비대화를 방지할 수 있다

## 12. 향후 고려 (v0.2+)

- 공유 대상 커스터마이징 (`cyolo profile share <add|rm> <item>`)
- `cyolo profile doctor` — 깨진 symlink 진단 및 자동 복구
- Shell completion (`cyolo <TAB>`, `cyolo profile <TAB>`)
- 설치 시 기존 `cyolo` 함수 자동 마이그레이션 체크
- `cyolo diet` 자동 실행 훅 (N일마다 자동)
- Git remote URL 기반 자동 프로필 매핑
